Instructions

You have been hired by a startup, BIN/CA$H, to do a security assessment of their new webapp. They have asked you to analyze the site for any security vulnerabilities which would let users take malicious actions.

You have been given a list of potential threats, which are tracked on the left side of the screen. Your goal is to demonstrate how an attacker could perform each attack, so the company knows how to fix them.

You have not been given permission to attack or deface the site outside of the attacks given to you. You should avoid any malicious actions that are not part of the exercise.

Here are the attacks they want you to demonstrate:

SQL Injection

View the "private" profiles which are not normally visible

SQL UNION Attack

Find out what Alice's balance is Hint: This attack is a slightly more advanced version of the attack in Lab 3. You may want to refer back to this guide from that lab. Hint: Each user's balance is stored in a column named "balance". The goal will be marked as complete once Alice's balance appears anywhere on the page. Profile names are case-sensitive.

Client-side security

Send a transaction which steals money from another user

CSRF

Create a local HTML file which exploits CSRF to send you a transaction Hint: If your attack is successful, this goal will be marked as "complete" when you open the file. It will work even though you are not allowed to send yourself a transaction. Note: Some folks have been having a problem where their CSRF requests do not include cookies. If you get redirected to the login page, make sure you are opening your attack in the same browser as a logged-in session. You may need to try using another browser.

Stored XSS

Trick the support staff into sending you money Hint: JavaScript code can be used to click buttons on the page. You can test code in the console before using it in an attack. Hint: The support staff will click any link you send them. You may need to refresh your page for the goal scoreboard to update.
